# LLMs Can Code, But Can They Do Design?

This repository contains the experimental data and results for the thesis **"LLMs Can Code, But Can They Do Design?"**. The purpose of this study is to evaluate the ability of Large Language Models (LLMs) to generate correct UML class diagrams from textual software requirements.

## Authors

This work was conducted by two team members. The complete set of requirements was divided equally between the authors, and each requirement was independently analyzed and evaluated following the same methodology.

## Objective

The goal of this research is to investigate whether modern LLMs are capable of performing software design tasks, specifically generating UML class diagrams from requirements specifications.

## Evaluated Models

The following models were evaluated:

- ChatGPT 5.5
- Claude Opus 4.8
- Microsoft Copilot
- Gemini 3.5 Flash

## Methodology

For each requirement:

1. A textual requirement specification is provided.
2. A manually designed UML class diagram is created as the reference solution (ground truth).
3. Each LLM is prompted to generate PlantUML code for the corresponding class diagram.
4. The generated PlantUML code is rendered into a UML class diagram.
5. The generated diagrams are compared with the manually created solution.
6. The following are identified:
   - Correct elements
   - Missing elements
   - Incorrect elements
   - Extra elements
7. Evaluation metrics are computed based on these counts.

## Repository Structure

Each requirement is stored in a separate folder.

Example:

```
Requirement_1/
│
├── 1.txt                 # Requirement specification
├── 1.png                 # Ground truth UML class diagram
│
├── 1.chatgpt.txt         # PlantUML generated by ChatGPT 5.5
├── 1.chatgpt.png         # Diagram rendered from ChatGPT output
│
├── 1.claude.txt          # PlantUML generated by Claude Opus 4.8
├── 1.claude.png          # Diagram rendered from Claude output
│
├── 1.copilot.txt         # PlantUML generated by Microsoft Copilot
├── 1.copilot.png         # Diagram rendered from Copilot output
│
├── 1.gemini.txt          # PlantUML generated by Gemini 3.5 Flash
└── 1.gemini.png          # Diagram rendered from Gemini output
```

The same structure is repeated for every requirement.

## File Description

| File            | Description                                  |
| --------------- | -------------------------------------------- |
| `n.txt`         | Textual requirement specification            |
| `n.png`         | Reference UML class diagram (ground truth)   |
| `q.<model>.txt` | PlantUML code generated by an LLM            |
| `n.<model>.png` | UML diagram rendered from generated PlantUML |

where `<model>` is one of:

- `chatgpt`
- `claude`
- `copilot`
- `gemini`

## Evaluation

Each generated diagram is manually compared against the reference solution.

The following quantities are recorded:

- **Correct elements**: elements that match the reference solution.
- **Missing elements**: elements present in the reference solution but absent from the generated diagram.
- **Incorrect elements**: elements generated incorrectly.
- **Extra elements**: additional elements that do not exist in the reference solution.

## Evaluation Metrics

### Recall

Measures how many elements from the reference solution were correctly generated.

```
Recall = Number of Correct Elements / Number of Elements in the Reference Solution
```

### Precision

Measures the proportion of generated elements that are correct.

```
Precision = Number of Correct Elements /
            (Number of Correct Elements + Number of False Elements)
```

where false elements include incorrect and extra elements.

### Over-Specification (OVS)

Measures how many unnecessary elements were introduced.

```
OVS = Number of Extra Elements /
      Number of Elements in the Reference Solution
```

## Purpose

This repository accompanies the thesis:

> **LLMs Can Code, But Can They Do Design?**

and contains:

- Requirement specifications
- Reference UML class diagrams
- PlantUML code generated by four LLMs
- Rendered diagrams
- Experimental data used for evaluation
- Metrics used to assess diagram quality

The repository provides all artifacts necessary to reproduce the experiments and compare the software design capabilities of current Large Language Models.
